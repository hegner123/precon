## Project: precon (March 16–17)

293 topics across 3 sessions spanning 2 days of active development. The March 16 session (101 topics) covered Phase 5 implementation and Phase 6 planning. The March 17 sessions (192 topics) covered a full remediation cycle and continued integration work.

### Pattern 1: File Surgery as Primary Edit Workflow (High Friction)

**Frequency: 8 of 10 complex edits used split/splice/mv instead of targeted tools**

The dominant code modification pattern across the March 17 sessions:

1. Write content to `/tmp/file.go`
2. `notab` to verify indentation
3. `tabcount` to double-check
4. `split` source file at target lines
5. `splice` tmp content into part 1
6. `splice` part 3 onto part 1
7. `mv` part 1 back to original
8. `delete` parts 2 and 3
9. `delete` the tmp file

Topics 113–122 and 125–145 both show this full sequence for inserting ~30 lines into `repl.go`. Each operation required 6–9 tool calls. The same change could have been accomplished with `splice` in insert mode — which already exists in the toolset and was used correctly at Topic 127 (`splice` with `line` parameter). The split/reassemble pattern was used anyway.

**Actionable:** When inserting into a file at a known line, `splice` with `{"mode":"insert","line":N}` is sufficient. The split-reassemble pattern adds 6 unnecessary steps and creates temporary state that can fail mid-operation.

### Pattern 2: Unnecessary File Reads Before Writing

**Frequency: Multiple instances across both days**

Several `write` operations were preceded by full reads of the file being replaced:

- March 16, Topic 22: Full read of `repl.go` (769 lines) → Topic 23: Complete rewrite of `repl.go`
- March 16, Topic 35: Full read of `precon.go` → Topics 36–39: Four targeted `repfor` patches
- March 17: All 9 recon tool calls in the duplicated Phase 5 pass (Topics 63–78) reproduced information already in context from the first pass earlier that same session

The `repl.go` full-read before a complete rewrite was defensible. The `precon.go` read before targeted patches was not — `sig` (used correctly at Topic 67 of the second pass) would have been sufficient. The second-pass recon was entirely wasteful.

**Actionable:** Use `sig` before targeted patches; reserve full `read` for complete rewrites. Before any recon sequence, verify the information isn't already in context from the current session.

### Pattern 3: Repeated Indentation Verification Ritual

**Frequency: Every file surgery sequence included notab + tabcount, even when both returned 0 changes**

Topics 113–115 show the pattern clearly:

- Write tmp file → `notab` → output: `"replacements":0` → `tabcount` → confirms correct indentation → proceed

Both tools consistently returned "no changes needed." The ritual was performed regardless. This costs 2 tool calls per edit without finding any issues across the entire session.

**Actionable:** Run `notab` and `tabcount` only when a build fails with whitespace errors, or when editing files that were manually created outside the REPL. Don't run them proactively on every splice operation.

### Pattern 4: Phase Confusion — Phase 5 Implemented Twice

**Critical finding: ~20 topics wasted re-implementing completed work**

- March 16, Topics 3–54: Full Phase 5 implementation — rewrote `persister.go`, `repl.go`, `adapter.go`, wired `main.go`, wrote 15+ tests, all 328 passing.
- March 16, Topics 59–80: Asked "lets start building phase 5 of PLAN.md" again. The assistant re-read the same files (`persister.go`, `repl.go`, `sqlite.go`, `client.go`, tier types), planned Phase 5 from scratch, and was preparing to re-implement it — until `:q` at Topic 81.

The second pass consumed ~20 topics in sig/read calls re-exploring code written that same session. The `precon` system is intended to solve exactly this problem, but it wasn't tracking the session's own meta-state.

**Actionable:** After a major phase completes, write a brief checkpoint to `PROGRESS.md`. A completed-phase summary should be the last thing written before any break in a session.

### Pattern 5: Code Quality Debt Accumulates Then Requires a Dedicated Session

**Frequency: 1 of 3 sessions was entirely remediation work**

The March 17 morning session (Topics 1–28) was entirely a code review and fix cycle addressing issues introduced during the March 16 build session. 12 weaknesses were identified, 10 fixed. Two were deferred: the `repl.go` god file and test coverage.

The deferred items are the same ones acknowledged as "not urgent but will become painful." The `repl.go` size was flagged twice — once during Phase 6 integration and once during the review. The file gained functions each phase without structural splitting.

**Actionable:** The `repl.go` split has been identified as needed twice. The split points are clear: command handling → `commands.go`, persistence logic → `persist.go`, output formatting → `render.go`. This will only get harder as phases 7+ add more to the file.

### Pattern 6: Verification Gaps After Multi-Step Edits

**Frequency: 3+ instances where a build/test was not run immediately after file modification**

Topic 100 (March 17): `write` to `main.go` returned an error ("file was modified after last read"). The tool call failed, but the session continued through Topic 101 (re-read) and Topic 102 (rewrite) without running `go build`. No build was run until Topic 129 — after 7 more tool calls modifying other files. The build passing at Topic 129 confirmed everything was fine, but a 7-step gap between write and verify is a window for silent failures.

**Actionable:** Run `go build ./...` after every write or splice that touches a `.go` file. Don't chain modifications across multiple files without verifying each compiles.

### Pattern 7: Stale File Read → Write Failure

**Frequency: 1 explicit failure (Topic 100), likely avoided other times**

Topic 100 shows the write tool rejecting a write because `main.go` had been modified between the last read and the write attempt. The `sig` tool had read the file at Topic 40; multiple other tools ran before the write at Topic 100. The fix was to re-read (Topic 101) then write (Topic 102).

This is a direct consequence of Pattern 6 — long chains of operations without verification create stale read windows.

**Actionable:** Before any `write`, the most recent interaction with that file should be a `read` (not `sig`, not `checkfor`). If more than 5 tool calls have happened since the last `read`, re-read before writing.

### Pattern 8: Tool Write Failure on Special Characters in Payload

**Frequency: 1 failure (March 16, Topic 24)**

A `write` call failed with `file_path is required`. The content was the full `main.go` (7,000+ chars) with a complex system prompt containing backtick interpolation. The `file_path` field was dropped, likely due to the JSON payload being malformed by the backtick string.

Topic 25 retried with the system prompt moved to a `buildSystemPrompt()` function, eliminating the backtick issue. Recovery was clean.

**Actionable:** When a `write` payload contains string literals with backticks or other special characters, externalize the content to a named function before writing. Don't embed large string literals directly in write payloads.

### Pattern 9: Test Failure on SQLite Timestamp Precision

**Frequency: 1 failure (March 16, Topic 44)**

`TestStore_UpdateRelevance` failed:
```
sqlite_test.go:467: LastAccessedAt should be updated after UpdateRelevance
```

The test stored a memory with `time.Now()` then immediately called `UpdateRelevance` and compared timestamps.

## Project: precon — Knowledge Gaps

### Gap 1: Phase Completion State Not Tracked (Critical)

**Evidence: Topics 3–54 and 59–83 (Phase 5 built twice in one session)**

Topics 6–57 show a complete Phase 5 implementation: persister rewrite, adapter, 328 tests passing, binary built. Then at Topic 59, the same session asked to "start building Phase 5." Topics 62–83 show the session re-reading pre-Phase-5 versions of files that had already been modified.

The session consumed approximately 20 topics re-exploring already-completed work before the duplicate effort was interrupted. The root cause: no mechanism for the assistant to verify current phase state before starting work.

**What would help:** Add `PROGRESS.md` to the project root. After each phase completes, write one line:

```
Phase 5: COMPLETE (2026-03-16) — persister tool_use, adapter, 328 tests passing
```

Before any "let's start Phase N" prompt, check `PROGRESS.md` first. This costs one `read` call and prevents 20-topic remediation cycles.

### Gap 2: Core Interface Re-Learning Each Session (High)

**Evidence: Topics 4, 67–69, 72, 80, 100 (repeated sig/read calls on the same files)**

`internal/tier/tier.go` was read via `sig` twice (Topics 67 and 100). `internal/api/types.go` was read twice (Topics 4 and 68). `internal/api/client.go` once (Topic 69). `internal/retriever/retriever.go` was read twice (Topics 5 and 72). `internal/storage/schema.go` was read twice (Topics 1 and 80). This is 6–8 redundant reconnaissance calls that reconstruct the same API surfaces each session.

**What would help:** Create `ARCHITECTURE.md` with a compact cheat-sheet of the four key interfaces — `tier.Store`, `tier.EvictionPolicy`, `persister.LLM`, `api.Client` — and their method signatures. ~50 lines would eliminate the repeated `sig`/`read` calls at session start.

### Gap 3: Tool Capability Awareness — `splice` Insert Mode (High)

**Evidence: Topics 113–128, 138–147 (file surgery performed twice in the same session)**

The split/splice/mv/delete pattern was used for every multi-line insertion, even though `splice` with `{"mode":"insert","line":N}` can accomplish the same result in a single call. Topic 128 shows the correct pattern was used once, but the full 8-step surgery sequence was repeated at Topics 138–147 immediately afterward.

This is not a one-time miss. The same session repeated the expensive pattern after already demonstrating the correct one.

**What would help:** Add a `splice` mode reference card to `CLAUDE.md` covering all three modes — `append`, `insert`, and `replace` — with a concrete example of when each applies.

### Gap 4: `repfor` Zero-Match Debugging (High)

**Evidence: Topics 95–99 and 111–123**

`repfor` returned `"files_modified":0` with no error. The session did not investigate why — trailing whitespace, Unicode characters, and line-ending differences are the three common causes — and immediately pivoted to full file surgery.

Each occurrence costs 3–5 topics of recovery. The gap is the absence of a systematic first step when `repfor` returns 0.

**What would help:** When `repfor` returns 0, first run `checkfor` with the same search string. If `checkfor` also returns 0, the string has been modified since last read — re-read the file and extract the exact current string before retrying. Do not proceed to surgery until the verbatim mismatch is confirmed.

### Gap 5: 7 of 13 Packages Have Zero Tests (High)

**Evidence: Topic 37 and Topic 3 (both show identical `[no test files]` for 7 packages)**

Packages with no tests, consistent across both observations: `api`, `dreamer`, `mcp`, `repl`, `retriever`, `synthesizer`, `pkg/precon`.

The highest-risk untested paths:
- `internal/repl` — the core loop, L1 pruning, and background persister wiring. Zero tests despite being 935 lines (Topic 54).
- `internal/api` — the HTTP client, retry logic, and streaming. Zero tests despite being the foundation of all external communication.
- `internal/retriever` — only 51 lines but completely untested.

**What would help:** The next gap-filling priority should be mock-based tests for `internal/api` and unit tests for `internal/repl` covering `pruneL1` and `persistTurn`.

### Gap 6: Write Payload Safety — Special Characters (Medium)

**Evidence: Topic 101 (failed write due to JSON payload corruption from backtick string)**

A `write` call failed with a malformed payload because the content included a raw string literal with backticks. The recovery at Topic 103 was to extract the backtick-containing content into a named function before writing.

**What would help:** Add to `CLAUDE.md`: "When writing files containing raw string literals with backticks, extract them into named `const` or function variables before writing."

### Gap 7: Build Verification After Tool Failures (Medium)

**Evidence: Topics 100–129 (29 topics between a write failure and the next build check)**

Topic 100: write failed. Topics 101–102: re-read and rewrote. Topics 103–128: modified 6 more files. Topic 129: first build verification. That is 29 topics between a confirmed write failure and verification that the system still compiled.

Write failures indicate state divergence. Subsequent writes may be operating on stale assumptions.

**What would help:** Any tool failure — write, splice, or `repfor` — triggers `go build ./...` before the next file modification.

### Gap 8: Project Layout Re-Discovery (Medium)

**Evidence: Topics 60 and 65 (stump called twice in the same session, returning identical 67-file trees)**

`stump` was called twice within the same session at depth=-1 and depth=2, both returning the same result. The project structure is stable enough that this information does not need to be reconstructed each session.

**What would help:** Add a compact directory summary to `CLAUDE.md` so `stump` calls are reserved for verifying changes rather than orienting at session start.

### Gap 9: SQLite Schema Migration Behavior (Latent Risk)

**Evidence: Topic 36 (role column present in live schema), Topic 23 (migration note in schema.go)**

The `role` column persisted in the live database after being removed from `schema.go` because SQLite's `CREATE TABLE IF NOT EXISTS` is a no-op when the table already exists. A migration note was added to the code, but the gap remains a latent risk for schema drift.

## Project: precon/test (March 17)

42 topics, 1 session. Two discrete tasks:

1. **Orientation** (Topics 1–15): Understanding an unfamiliar codebase
2. **Feature addition** (Topics 16–42): Adding a `DueDate` field to the task schema

### Pattern 1: Thorough Upfront Exploration Before Any Changes

Topics 6–14 show 9 consecutive read/sig operations before a single edit was made. Files read: `PLAN.md`, `START.md`, `task.go`, `store.go`, `server.go`, `config.go`, `store_test.go`, `server_test.go`, `valid.json`, `invalid.json`, `server.go` (full).

**This is a strength, not an inefficiency.** But watch for over-indexing: `server.go` and `server_test.go` were read in full for a change that never touched the server layer.

### Pattern 2: Change Scope Was Correctly Identified and Executed

The `DueDate` addition touched exactly the right files:
- `model/task.go` (struct + validation)
- `store/store.go` (UpdateTask copy)
- `migrations/001_init.sql` (schema reference)
- `testdata/valid.json` (fixture)
- `store_test.go` (4 new tests)

No file was modified unnecessarily. The SQL migration also caught a pre-existing gap (`category` column was missing from the original schema) and fixed it proactively.

### Pattern 3: Test Data Bug Introduced in `valid.json`

Topic 35 added `"due_date": "2026-12-31T00:00:00Z"` to `valid.json`. The validation in `task.go` rejects past dates. `2026-12-31` passes now but this fixture will silently rot — by January 2027, `make seed` will start failing.

**Fix:** Either use a relative date in the seed script or remove the `due_date` from the fixture and test it only in unit tests with `time.Now().Add(...)`.

### Pattern 4: Temp File Approach for Test Insertion Is Unusual

Topics 38–39 wrote the new tests to `/tmp/due_date_tests.go` and then used `splice` to insert them into `store_test.go`. This works but it's indirect — a direct append to the target file would be simpler.

### Pattern 5: No Verification of the Seed Script Against the New Field

The session verified via `go build ./...` and `go test ./... -v` (Topics 40–41, all 21 tests pass). But `scripts/seed.sh` and `scripts/smoke.sh` were never read or tested. Given `valid.json` was modified and the seed script presumably uses it, this is an untested surface.

### Knowledge Gaps (test project)

| Gap | Confirmed / Inferred | Suggested Remedy |
|---|---|---|
| Fixture date hygiene after time-based validation | **Confirmed** | Personal checklist: audit fixtures after adding temporal validation |

## Project: dealeredge-demo-generator (March 17–18)

### Pattern 1: Branch Confusion Is a Recurring Friction Point

Happened twice in a single session:

- **First instance:** You were on `master`, needed to switch to `staging` to access `DEV1-API-BACKEND-INSTRUCTIONS.md`. The file was found only after branch investigation.
- **Second instance (Topics 53-61):** After switching to `staging`, the DEV1 doc still wasn't accessible — because it's actually on `doc-test`, not `staging`. Required 4 tool calls to locate it.

The `doc-test` branch name is genuinely confusing — it has production SMS infrastructure code with a commit message "documentation test."

**Concrete cost:** ~6 tool calls and significant session time spent on branch navigation.

### Pattern 2: Memory-vs-Disk Confusion (Injected Context vs. Actual Files)

Topics 62-69 reveal a recurring ambiguity: precon injects summaries as context, but those summaries describe files that don't exist on the current branch or disk.

The agent read injected context as a file reference, attempted `read` on a nonexistent path, then spent 3 tool calls searching for it before concluding it was memory-only.

**Structural problem:** Precon's memory summaries look like file references but aren't always backed by files on the active branch. Topics 32-45 show the same pattern in reverse — you explicitly asked Claude to read docs "to get them into precon's memory," treating file reads as a memory-loading mechanism.

### Pattern 3: Documentation Reads as Primary Workflow

5 of 7 sessions involved reading documentation rather than writing code. Topics 30-45 were explicitly a doc-ingestion sweep (13 files read). Topics 75-85 were a full project assessment. No code was written in either session.

### Pattern 4: Tool Mis-Fires on .js Files

Topics 12-13: `sig` was called on `mcp-server/lib/registry.js` and `mcp-server/lib/helpers.js` and errored both times — `sig` doesn't support `.js` files. Then `read` was used as the fallback. For a JS-heavy codebase, this will happen repeatedly.

### Pattern 5: Project Complexity Accurately Documented But Underweighted

The assessment correctly identifies core risks: 5 products in one repo, fragmented state management, heavy dependency weight, comms system unshipped despite mature specs. But the `doc-test` branch with critical SMS infrastructure hasn't been merged to `staging` or `master`. Documentation discipline is high, but the merge/integration cadence may not match it.

### Actionable Recommendations

1. **Rename `doc-test` immediately.** Call it `feat/sms-infrastructure` or `feat/comms-phase1`.
2. **Create a `BRANCHES.md`** listing active branches and what's on each. With 30+ branches, this is no longer obvious.
3. **Establish a rule for memory-backed context:** Note in summaries whether content is backed by a file path on a specific branch.
4. **Use `read` directly for `.js` files** — skip `sig` for files outside `.ts/.tsx/.go/.cs`.
5. **The next session should write code, not read docs.** The context is loaded. The specs are validated.
